# 精益厨房 V3 — 产品定义

> 精益厨房，即通过对一道具体菜的数字化分解，来高效实现精准采购、切配等后厨高效协作，来满足以炒菜机器人生产模式下，人机协同的智能后厨精益管理。
>
> ——浩哥

---

## 一、一句话定义

**精益厨房是一套面向炒菜机器人生产模式的智能后厨管理系统。核心是对菜品做数字化分解，驱动精准采购与切配，实现人机协同下的高效后厨管理。**

---

## 二、目标用户

### 3 类用户

| 角色 | 核心任务 | 使用端 |
|------|---------|--------|
| **厨师长** | 菜品排程。安排每日要做的菜品及份数，系统自动分解为采购计划单和切配工单 | PC 浏览器 |
| **采购员** | 采购管理。录入采购单（含拍照识别），费用报销（后续阶段）。移动端操作 | **手机 PWA**（添加到主屏幕，类原生体验） |
| **运营人员** | 围绕菜品的所有数据管理。食材字典、菜品档案、基础数据的维护 | PC 浏览器 |

**当前不做复杂的角色权限体系。** 移动端只有采购员可用（采购管理 + 库存查看），PC 端由厨师长和运营人员共用。

---

## 三、核心功能

### 6 大功能模块

#### 1. 排程管理（业务起点）
厨师长创建排程（选择日期、菜品及份数）→ 系统自动分解为采购计划单和切配工单。
- 排程以"天"为单位
- 状态：**进行中 / 已完成**（两个状态）
- 支持查看历史排程
- 分解结果一目了然：需要买什么、需要切什么

#### 2. 采购管理
采购管理与排程计划**不关联**，两者独立。采购员根据实际采购情况录入。

**采购单录入（移动端）：**
- 采购员拍照上传采购单
- 系统调用大模型识别图片，自动提取物料品名、数量、单价等
- 提取后填入表单，采购员确认或修改
- 确认即入库

**费用报销：** 当前阶段不做，待采购单管理跑通后再处理。

#### 3. 库存管理
采购单录入即代表入库，切配工单完成则代表食材出库，库存自动变动。
- 需要清晰知道当前每种食材的库存数量
- 原料走精确出入库管理
- 小料、调料、酱料按**备货制**管理，不追踪单次消耗
- **调料库存当前阶段暂不处理**（小料/酱料同）

#### 4. 食材库
食材包括：
- **原料**：采购的基本食材（猪肉、青椒等）
- **净料**：原料经初加工后的规格化半成品（肉片、肉丝等）
- **小料**：姜蒜葱花等（备货制）
- **调料**：生抽、盐、鸡精等（备货制）
- **酱料**：复合调味酱（备货制）

实现基本增删改查。

#### 5. 菜品库
一道菜的完整档案记录。是实现菜品分解的唯一依据。
- 包含：基础信息 + 5 类食材配方（主料/辅料/小料/调料/酱料）+ 加工流程
- 实现基本增删改查

#### 6. 数据字典
菜品类别、食材分类、单位、供应商等基础数据的维护。

## 四、数据体系

```
菜品档案（一切起点）
    ├─ 原料 → 采购库存体系（精确管理）
    │       菜品配方中的原料 → 自动生成采购需求 → 采购入库 → 库存
    │
    ├─ 净料 → 菜品分解依据
    │       菜品配方中的净料规格 → 排程分解为切配工单
    │
    ├─ 小料 → 备货制（月度盘点）
    ├─ 调料 → 备货制（月度盘点）
    └─ 酱料 → 备货制（月度盘点）
```

**每个操作必须记录操作人和操作时间。**

---

## 五、不做什么

- 不做无意义的统计报表
- 不做复杂的角色权限体系（当前阶段）
- 不做非必须的经营分析
- 不做数据分析大屏
- 不做移动端以外的复杂交互
- **不做费用报销**（当前阶段，采购单管理跑通后再处理）
- **不做调料/小料/酱料的精确库存**（备货制，当前只管理原料库存）

只做最实用、非做不可的功能。

---

## 六、产品原则

1. **菜品档案是一切起点。** 原料对应采购库存体系，净料对应一道菜的分解。没有菜品档案，就没有排程分解，就没有精准采购。
2. **每个动作必须记录操作人和操作时间。** 所有增删改操作都带有追溯信息。
3. **不对齐不推进。** 不确定的需求不猜，确定了再做。
4. **先跑通再优化。** 第一阶段目标是一条完整链路跑通——排程→采购→切配→入库→出库。链路跑通后，再优化体验。

## 七、技术方案要点

| 维度 | 方案 |
|------|------|
| 部署 | Zeabur（同时托管 Next.js 应用和 PG 数据库） |
| 前端框架 | Next.js + shadcn/ui |
| 数据库 | PostgreSQL |
| 图片识别 | 移动端上传 → 服务端调用多模态大模型识别 → 返回结构化数据 → 用户确认 |
| 采购端方案 | **PWA（Progressive Web App）**。采购员在手机浏览器访问 → 添加到主屏幕 → 获得类原生体验（全屏、离线缓存、推送通知）。不需要走应用商店审核，更新即时生效 |
| 拍照上传 | 浏览器原生 API（`<input type="file" accept="image/*">`），不需要原生 App 能力 |

---

## 八、PWA 方案说明

采购员端的核心交互是**拍照上传采购单**和**查看库存**，不需要原生 App 的复杂能力。PWA 是最优选择：

| 维度 | PWA | 原生 App |
|------|-----|---------|
| 开发成本 | 与 Web 系统共用代码，一套代码 PC + 手机 | 需要单独开发维护 |
| 安装流程 | 浏览器打开 → 添加到主屏幕 → 可用 | 应用商店审核 → 下载安装 |
| 更新 | 刷新即更新，无需审核 | 发版 → 审核 → 用户手动更新 |
| 拍照能力 | 浏览器 Camera API 支持 | 原生 API |
| 离线能力 | Service Worker 缓存部分数据 | 全离线 |
| 使用场景匹配 | ✅ 现场采购员正好适用 | 过度 |

采购员使用流程：
1. 手机浏览器打开系统地址
2. 浏览器提示"添加到主屏幕"（或手动添加）
3. 下次从桌面图标直接打开，全屏无地址栏，像原生 App
