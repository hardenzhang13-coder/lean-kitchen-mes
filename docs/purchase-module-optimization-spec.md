## 二、详细需求说明

### 需求1：AI 识别食材名称匹配逻辑 — 改为完全匹配

#### 现状问题
当前 AI 识别后的食材匹配逻辑使用了"部分包含"匹配（`includes`），导致识别准确性不足。例如，食材库中存有"生抽"，AI 识别出"海天金标生抽"时，因包含"生抽"而被错误匹配。

#### 目标业务逻辑
AI 识别提取的 `ingredientName`（从 `productName` 中剥离品牌前缀后的食材名称）与食材库中原料的 `name` 或 `alias` 字段、以及调料的 `name` 字段，必须进行**字符串完全相等**匹配。

- 原料匹配：仅当 `ingredientName === ingredient.name` 或 `ingredientName === ingredient.alias` 时匹配成功
- 调料匹配：仅当 `ingredientName === seasoning.name` 时匹配成功
- 不满足完全匹配条件的，一律视为"未匹配"
- 未匹配项在录入页表格中以红色高亮（`bg-red-50/50`）展示，并保留"新增食材"按钮

#### 涉及范围
AI 识别接口（`/api/purchase-receipts/recognize`）中的原料匹配和调料匹配逻辑。

> 注意：完全匹配后未匹配项比例可能上升，需确保未匹配项的"新增食材"弹窗体验流畅，不影响录入流程。

### 需求2：已录入采购单重复识别业务逻辑

#### 目标
防止同一张采购单图片被重复识别并录入系统，减少数据冗余。

#### 业务逻辑
1. **重复检测机制**：在 `PurchaseReceipt` 表中新增图片标识字段（如 `imageHash`），用于标识已上传的采购单图片。当用户上传图片进行 AI 识别时，系统先计算该图片的特征标识，并检查是否已有已录入（非作废）的采购单使用了相同图片。
2. **检测到重复时的交互**：
   - 在 AI 识别返回结果前，若发现已有相同图片的采购单，向用户展示提示信息："检测到该采购单图片已录入（采购单 #XXX），是否继续识别？"
   - 提供两个选项：
     - **"查看已有采购单"**：直接跳转到该采购单详情页
     - **"继续识别"**：清空当前页面数据，用新识别结果覆盖，允许用户重新录入
3. **行为约束**：重复检测为提示性，不强制阻止用户录入。用户可能因单据更新而有意识重新录入。

#### 涉及范围
- 采购单录入页（`/purchases/new`）的图片上传与识别流程
- AI 识别接口（`/api/purchase-receipts/recognize`）的请求/响应结构
- 采购单数据模型（新增图片标识字段）

### 需求3：新增采购去向/部门字段（业务术语"采购单位"）

#### 目标
在采购单上标记该批采购物资的归属部门/去向，便于后续按部门统计和管理。

#### 字段说明
- **业务名称**：采购单位（注意：此处的"采购单位"指"采购归属部门/去向"，与明细行中的"采购计量单位"（斤/件/箱）是完全不同的概念）
- **建议数据库字段名**：`purchasingUnit`（`String`，非空，带默认值）
- **可选值**：
  1. 切配中心（默认值）
  2. 新京企业食堂
  3. 印象京山会所
  4. 船九

#### 交互规范
- **录入页（`/purchases/new`）**：
  - 位置：放在"采购日期"字段下方，"供应商"字段上方
  - 组件：使用 `TileGroup` 平铺单选（4个选项，符合系统"≤5 选项用 TileGroup 平铺"的交互规范）
  - 必填：是，未选择时提交校验失败
- **编辑页**：可修改，与录入页展示一致
- **列表页（`/purchases`）**：
  - 位置：在"状态"列之后展示该字段
  - 展示方式：纯文本
- **详情弹窗**：在基础信息区域增加"采购单位"行

#### 数据模型变更
`PurchaseReceipt` 表新增字段 `purchasingUnit`（`String`，非空，默认值为 `"切配中心"`）。

### 需求4：将"更换采购单"操作改为"清空数据"操作

#### 目标
将当前图片上传区的"更换采购单"按钮改为"清空数据"按钮，语义更明确，允许用户完全重置录入状态。

#### 业务逻辑
1. **按钮文案**：由"更换采购单"改为"清空数据"
2. **按钮样式**：保持为 secondary/outline 样式，与"AI 识别"按钮并排
3. **点击后行为**：
   - 弹出二次确认弹窗："确定清空所有已录入数据吗？此操作不可撤销。"
   - 确认后，清空当前页面所有已识别/录入的内容：
     - 采购明细（`items`）全部清空
     - 供应商、摘要、采购总金额清空
     - 采购去向/部门（`purchasingUnit`）重置为默认值
     - 已上传的图片清空（`imageBase64`、`imagePreview`）
     - 页面回到初始空状态，用户可重新上传新图片并识别
4. **编辑模式例外**：当页面处于编辑模式（URL 中有 `?id=xxx`）时，不显示"清空数据"按钮。编辑模式下的操作是修改已有采购单，不应允许一键清空。

#### 涉及范围
采购单录入页（`/purchases/new`）的左侧图片上传区域与基础信息区域。

### 需求5：采购模块列表组件样式更新为系统最新标准

#### 目标
将采购单列表页（`/purchases`）的样式统一为系统当前最新标准，参照已交付的排程列表（`/schedules`）和食材列表（`/ingredients/raw`）的规范。

#### 具体更新项

1. **页面头部**：使用 `PageHeader` 组件（标题 + 描述），替代当前手写标题
2. **状态筛选**：
   - 当前使用 `Select` 下拉组件选择状态（待结算/已结算/已作废/全部）
   - 改为 Tab 切换形式：参照排程列表的 `bg-muted rounded-lg p-1` 的 Tab 切换栏，四个选项平铺展示
3. **筛选区布局**：统一为 `flex items-center gap-3 flex-wrap` 的筛选行布局，包含：搜索框、日期范围选择器、清除按钮
4. **加载状态**：当前使用 `div h-[300px] bg-muted rounded-md animate-pulse` 作为加载骨架，改为使用 `SkeletonTable` 组件（参照食材列表），列数与表格列数一致
5. **空状态**：当前使用 `FileText` 图标 + 居中文字，统一为排程列表/食材列表的标准空状态：图标 + 文字 + 引导按钮（"录入第一单"）
6. **分页**：当前列表无分页，参照食材列表接入 `Pagination` 组件 + `usePagination` hook
7. **表格列顺序调整**：加入"采购单位"列后，列顺序为：编号、状态、采购单位、采购单摘要、采购日期、供应商、总金额、操作人、创建时间、操作

### 需求6：采购明细编辑弹窗样式与字段优化

#### 目标
优化采购单录入页中采购明细行的编辑弹窗，使其更紧凑、更符合录入场景。

#### 具体优化项

1. **字段布局密度**：
   - 当前编辑弹窗使用 `FormSection cols={2}`（一行2个字段），改为 `FormSection cols={3}`（一行3个字段）
   - 字段顺序需合理安排，确保逻辑相关字段相邻（如采购规格-数量-采购单位在同一行，入库单位-入库数量-食材单价在同一行）

2. **弹窗宽度**：
   - 当前弹窗宽度为 `sm:max-w-[700px]`，对于3列布局过窄
   - 改为更宽的尺寸，如 `sm:max-w-[1000px]` 或 `sm:max-w-[1100px]`，确保3列布局不拥挤

3. **数量字段整数输入**：
   - 编辑弹窗中的"采购数量"（`qty`）和"入库数量"（`stockInQty`）输入框**只能输入整数**（不允许小数）
   - 输入组件去掉上下增减值操作（即隐藏 `type="number"` input 的 spinner 箭头）
   - 注意：此限制**仅针对编辑弹窗**，表格内直接输入的 `qty` 和 `stockInQty` 保持现状（允许小数），因为表格内快速修改可能需要更灵活

4. **只读字段展示**：
   - 编辑弹窗中的只读字段（食材ID、一级分类、二级分类）保持当前展示样式，但需适配3列布局（食材ID可占整行或合并列）

### 需求8：采购单位为"斤"时入库数量默认等于识别数量

#### 目标
当 AI 识别出的食材采购单位（`purchaseUnit`）为"斤"时，简化入库数量的默认值逻辑。

#### 业务逻辑
1. **识别后映射阶段**：在 AI 识别结果转换为前端 `FormItem` 时，当 `purchaseUnit === "斤"`：
   - `stockInQty` 的默认值直接等于 `qty`（识别出的数量），不再通过规格解析器计算或默认设为 1
   - `stockUnit` 的默认值设为 `"斤"`
2. **非"斤"单位**：当 `purchaseUnit !== "斤"` 时，保持现有逻辑（通过 `spec-parser` 计算或默认 1）
3. **用户可覆盖**：此默认值仅在 AI 识别后自动填充，用户仍可在表格中直接修改或在编辑弹窗中调整

#### 典型场景
- AI 识别出：猪肉 10斤 → 采购数量=10，采购单位=斤 → 入库数量默认自动填 10，入库单位默认=斤
- AI 识别出：生抽 1件 → 采购数量=1，采购单位=件 → 保持现有逻辑（通过规格如"1.9L*6"解析入库数量）

## 四、API 接口变更

### 1. POST /api/purchase-receipts/recognize（AI 识别）

**请求**：无变更（仍接收 `imageBase64`）

**逻辑变更**：
- 新增：检查 `imageHash` 是否已有已录入的采购单（非作废状态），若存在则返回 `duplicateWarning` 信息（含已有采购单ID）
- 变更：原料和调料匹配改为**完全匹配**（`===` 字符串相等），移除 `includes` 部分匹配逻辑
- 新增：当 `purchaseUnit === "斤"` 时，`stockInQty` 默认值 = `qty`，`stockUnit` 默认值 = `"斤"`

**响应结构变更**：
```json
{
  "items": [...],
  "summary": "...",
  "totalAmount": 123.45,
  "supplierId": 1,
  "supplierName": "...",
  "duplicateWarning": {        // 新增，可选
    "receiptId": 123,
    "message": "检测到该采购单图片已录入（采购单 #123）"
  }
}
```

### 2. GET /api/purchase-receipts（采购单列表）

**响应变更**：
- 返回字段新增 `purchasingUnit`
- 新增分页参数支持（`page`、`pageSize`，后端分页查询）

### 3. GET /api/purchase-receipts/[id]（采购单详情）

**响应变更**：
- 返回字段新增 `purchasingUnit`

### 4. POST /api/purchase-receipts（创建采购单）

**请求变更**：
- 接收并校验 `purchasingUnit`（必填，枚举值之一）
- 接收并保存 `imageHash`（如果上传了图片）

### 5. PUT /api/purchase-receipts/[id]（编辑采购单）

**请求变更**：
- 接收并校验 `purchasingUnit`（必填，枚举值之一）
- 已有逻辑（编辑时回滚库存→更新数据→重新入库）保持不变

## 六、技术约束与红线

1. **禁止新增 `any` 类型**：所有新增/修改代码必须使用严格类型，现有 `any` 不扩大使用
2. **禁止无分页的列表 API**：采购单列表必须支持服务端分页
3. **输入校验**：所有新增 API 参数必须通过 Zod schema 校验
4. **事务内操作**：涉及库存更新的操作（创建/编辑采购单）必须放在 `prisma.$transaction` 中
5. **操作日志**：新增/编辑/作废采购单操作需记录 `logOperation`
6. **Schema 迁移**：必须通过 `prisma migrate dev` 生成迁移文件，禁止直接 `db push`
7. **代码提交前检查**：必须按顺序执行 `npx tsc --noEmit`、`npm run build`、`npm run lint`，任一步失败不得提交
