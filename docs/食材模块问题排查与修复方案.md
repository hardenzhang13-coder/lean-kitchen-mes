## 问题1：小料分类种子数据无法删除

### 根因
1. **前端不检查 res.ok**：`app/settings/classes/page.tsx` `handleDelete` 只 `await fetch(...)`，不检查 `res.ok`。fetch 不会因 HTTP 400 抛错，后端外键约束失败时前端仍显示"删除成功"。
2. **后端未检查引用**：`DELETE /api/ingredient-categories/[id]?type=l2` 直接 `prisma.ingredientCategoryL2.delete()`，未检查该 `l2Code` 是否被 `ingredients` 或 `net_ingredients` 引用。

### 修复点
- **1A**（前端）：`app/settings/classes/page.tsx` `handleDelete` 增加 `res.ok` 检查，非 ok 时抛错：`const res = await fetch(...); if (!res.ok) { const err = await res.json().catch(() => ({})); throw new Error(err.error || "删除失败"); }`
- **1B**（后端）：`app/api/ingredient-categories/[id]/route.ts` `DELETE` 的 `type === "l2"` 分支，删除前查 `prisma.ingredient.count({ where: { l2Code: row.code } })` 和 `prisma.netIngredient.count({ where: { l2Code: row.code } })`，任一 > 0 返回 400 `{ error: "该分类已被食材引用，无法删除" }`。
- **1C**（后端）：`type === "l1"` 分支删除前，检查所有子分类的 `l2Code` 是否被引用，同上。

## 问题2：净料弹窗选择来源信息优化（3个子问题）

### 根因
- **2.1 分类样式混乱**：`app/components/ingredient-selector-dialog.tsx` 第 127-185 行，一级分类未选中时 `bg-transparent border-input`，与输入框背景混为一体；二级分类默认显示 EmptyState（"请选择一级分类"），视觉断裂。
- **2.2 未选分类不能选食材**：第 204 行 `handleConfirm` 强制 `if (!selectedIngredientId || !selectedL2Code) return;`，用户希望通过搜索直接选食材，不想先选分类。
- **2.3 更换原料按钮**：`app/components/net-ingredient-form-fields.tsx` 第 141 行使用"更换原料"按钮，用户希望改为来源信息卡片右上角 X 清空图标。

### 修复点
- **2A**（分类样式）：`ingredient-selector-dialog.tsx` 一级/二级分类按钮统一使用 TileGroup 风格（圆角、明确边框、统一 padding），未选中与选中态仅在颜色上区分，不再使用 `bg-transparent`。
- **2B**（默认状态）：一级分类默认展开，二级分类直接展示所有可用分类，不显示 EmptyState。或：允许不选一级分类时直接搜索全部食材。
- **2C**（无分类选择）：`handleConfirm` 改为只检查 `selectedIngredientId`：`if (!selectedIngredientId) return;`。`l2Code` 从所选食材的 `l2Code` 属性自动推导，`l1Code` 从 `categories` 反查父级。
- **2D**（清空ICON）：`net-ingredient-form-fields.tsx` 第 120-145 行，来源信息展示区（三列 div）的右上角增加 `<X className="h-4 w-4" />` 图标按钮，点击清空 `sourceIngredientId`/`l1Code`/`l2Code`，替代"更换原料"按钮。

## 问题5：转净料操作弹窗优化（3个子问题）

### 根因

- **5.2 展示/输入样式不区分**：`app/components/net-ingredient-form-fields.tsx` 第 127-140 行，展示字段用 `bg-muted`，与 Input 背景色接近，视觉上难以区分只读和可输入。
- **5.3 placeholder 未消失**：来源信息展示字段空状态时显示 `"—"`，用户期望类似 placeholder 的提示文案 `"请选择来源原料"`（灰色）。

### 修复点
- **5A**（保存关闭）：`net-ingredient-form-dialog.tsx` 第 287 行，条件改为 `if (initialData?.id || isMinor || isConvert) { onOpenChange(false); onSuccess?.(); } else { resetForCreate(); ... }`。
- **5B**（样式区分）：`net-ingredient-form-fields.tsx` 第 127-140 行，展示字段 `div` 改为 `border border-dashed border-muted-foreground/30 bg-muted/50 text-muted-foreground`，与实线边框的 Input 形成明确区分。或使用 `FormField` 的 `readOnly` 统一样式。
- **5C**（空状态文案）：展示字段空状态时显示 `"请选择来源原料"`（`text-muted-foreground`），替代 `"—"`。

## 问题6：净料首页统计与实际不符 + 筛选缺失 + 视觉统一

### 根因
- **6.1 统计不符**：`app/ingredients/page.tsx` 第 57 行：`prisma.netIngredient.count({ where: { deletedAt: null, l2Code: { notIn: minorL2Codes } } })`。`getMinorL2Codes()` 查询 `parentCode: "MIN"`。如果 `minorL2Codes` 为空数组（MIN 分类未 seed 或数据异常），`notIn: []` 在 Prisma 中匹配所有记录，统计值可能包含全部净料数据（包括小料），导致"净料"和"小料"统计重叠或不准确。需要确认 `deletedAt` 字段的默认值和 `l2Code` 的 null 值处理。
- **6.2 缺少筛选**：`app/ingredients/net/page.tsx` 第 110-119 行，只有关键词搜索，没有一二级分类筛选（对比原料页面有 `l1Filter`/`l2Filter`）。
- **6.3 视觉不统一**：净料列表没有 `CategoryTag` 组件展示二级分类（原料页面使用 `CategoryTag`），且没有分类筛选胶囊（Badge/pill）。

### 修复点
- **6A**（统计逻辑）：`app/ingredients/page.tsx` 第 57 行，当 `minorL2Codes.length === 0` 时，`notIn` 条件应改为 `AND: [{ deletedAt: null }, { l2Code: { not: null } }]`，避免空数组导致全匹配。或者更简洁：不使用 `notIn`，改用 `l2Code: { not: null }` 兜底。确保小料和净料的统计互斥且覆盖全部。
- **6B**（筛选功能）：`app/ingredients/net/page.tsx` 参照原料页面（`app/ingredients/raw/page.tsx` 第 108-139 行），增加 `l1Filter`/`l2Filter` state 和筛选逻辑：关键词搜索 + 一级分类筛选 + 二级分类筛选。
- **6C**（CategoryTag）：`app/ingredients/net/page.tsx` 表格列中增加二级分类列，使用 `CategoryTag` 组件展示（参照原料页面第 320-328 行）。
- **6D**（筛选胶囊）：净料页面顶部增加一级分类筛选下拉 + 二级分类筛选下拉，与原料页面保持一致。

## 问题7：原料/小料/酱料表格样式统一

### 根因
各页面表格列数、空状态、操作按钮排列不一致：
- 原料页面：使用 `DataTable` + `CategoryTag` + 一二级分类筛选 + 序号/编号/名称/分类/规格/价格等 10+ 列
- 小料页面：使用 `DataTable`，但缺少 `CategoryTag`、缺少分类筛选，列数 7 列（序号/二级分类/编号/名称/规格/单价/单位）
- 酱料页面：使用 `DataTable`，但缺少 `CategoryTag`、缺少分类筛选，列数 8 列（序号/编号/名称/品牌/规格/类型/单价/单位）
- 净料页面：使用 `DataTable`，但缺少 `CategoryTag`、缺少分类筛选，列数 7 列（序号/编号/名称/规格/来源原料/出成率/单价/单位）

### 修复点
- **7A**（小料表格）：`app/ingredients/minor/page.tsx` 表格增加 `CategoryTag` 展示二级分类（参照原料页面），增加一二级分类筛选（顶部筛选区）。
- **7B**（酱料表格）：`app/ingredients/sauce/page.tsx` 表格增加 `CategoryTag` 展示类型（如果适用），或增加分类筛选（如果酱料未来有分类体系）。当前至少统一搜索框样式和表格列间距。
- **7C**（净料表格）：`app/ingredients/net/page.tsx` 增加 `CategoryTag` 展示二级分类，增加一二级分类筛选，统一列宽和字体大小。
- **7D**（SkeletonTable 列数统一）：各页面 `SkeletonTable cols={N}` 应与实际表格列数严格一致。原料约 10 列，小料 8 列（增加 CategoryTag 后），酱料 8 列，净料 8 列（增加 CategoryTag 后）。
- **7E**（表格行高与字体）：统一使用 `DataTable` 标准样式，不额外覆盖行高或字体大小，确保 4 个页面视觉一致。

## 问题8：测试环境登录后显示"数据获取失败" + 退出无反应

### 根因
- **8.1 数据获取失败**：登录流程 `app/login/page.tsx` 第 31-38 行，`fetch("/api/auth/me")` 检查当前会话。如果返回非 ok，则不做 redirect，显示登录表单。用户说"登录后显示数据获取失败"，意味着登录表单提交后，页面跳转到工作台（`/`），但工作台的某个数据请求失败了。更可能的是：登录后 `router.replace("/")` 跳转到工作台，工作台中的 `fetchData`（如 `app/page.tsx` 的某个 API 调用）失败，显示"获取数据失败"的 toast。
  - 可能原因：测试环境数据库连接失败（`DATABASE_URL` 未配置或 PG 不可达），或某个 API 抛 500 错误。
- **8.2 退出无反应**：`app/components/sidebar.tsx` 第 89-92 行：
  ```typescript
  await fetch("/api/auth/logout", { method: "POST" });
  toast.success("已退出");
  router.push("/login");
  router.refresh();
  ```
  如果 `fetch("/api/auth/logout")` 因网络错误抛错（非 HTTP 错误，是网络层错误），整个 async 函数会抛错，但 `onClick` 没有 try-catch，错误未被捕获，后续 `toast.success` 和 `router.push` 不会执行。用户看到 toast 没出来，页面也没跳转。但实际上如果 API 返回 500，fetch 不会抛错，toast 会显示。所以更可能是网络层问题（测试环境域名/API 不可达）或 `router.push` 后页面没有实际刷新。
  - 另一个可能：`router.push("/login")` 后，`router.refresh()` 立即执行，如果页面在刷新过程中又被 redirect 回 `/`（因为 session 没清除），就会出现"已退出"toast 但页面仍在原地的现象。

### 修复点
- **8A**（登录错误排查）：先检查测试环境 `DATABASE_URL` 是否配置正确。`app/login/page.tsx` 登录成功后跳转到 `/`，如果工作台 API 报错，需要看具体哪个 API 报错。建议先在测试环境手动访问 `/api/auth/me` 和 `/api/ingredients` 等接口，确认数据库连通性。
- **8B**（退出健壮性）：`app/components/sidebar.tsx` 第 89-92 行，增加 try-catch 包裹 fetch 和路由跳转：
  ```typescript
  onClick={async () => {
    try {
      await fetch("/api/auth/logout", { method: "POST" });
      toast.success("已退出");
    } catch {
      toast.error("退出请求失败");
    }
    router.push("/login");
    router.refresh();
  }}
  ```
  确保即使 fetch 失败，也强制跳转到登录页。
- **8C**（退出后缓存）：`logout` API 后，添加 `window.location.href = "/login"` 强制整页刷新，避免 Next.js 客户端缓存导致页面状态残留。或者确保 `clearSessionCookie` 正确清除了 cookie 且 cookie 的 path 和 domain 与测试环境一致。
