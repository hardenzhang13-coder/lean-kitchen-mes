## 一、产品背景与目标

### 1.1 业务背景

净料（原料经初加工后的规格化半成品）是菜品配方（BOM）和切配工单的基础数据。当前系统净料表（`net_ingredients`）数据为 **0 条**，导致排程→切配链路无法运转。运营人员需要基于现有 259 种原料，快速创建对应的净料规格档案（如"带皮五花肉 → 去皮五花肉片/丝/块"）。

### 1.2 业务目标

| 目标 | 量化指标 | 验收方式 |
|------|----------|----------|
| 提升净料创建效率 | 单原料创建多个净料规格 ≤ 3 步/个 | 操作路径测试 |
| 保证成本数据准确性 | 净料单价自动计算率 100% | 自动计算覆盖率 |
| 建立原料→净料可追溯关系 | 每个净料必须关联源原料 | 数据校验 |

### 1.3 产品定位

在现有 `net_ingredients` 表基础上的**功能增强**，不做表结构重构，新增自动计算逻辑、复合弹窗交互、原料→净料快捷入口。

## 二、用户角色与场景

### 2.1 利益相关者

| 角色 | 使用场景 | 使用频率 | 使用端 |
|------|----------|----------|--------|
| **业务运营** | 维护净料档案（基于原料创建净料规格） | 低（初期批量创建，后续按需） | PC 浏览器 |
| **厨师长** | 查看净料规格以确认菜品配方可用性 | 中 | PC 浏览器 |

### 2.2 核心场景

**场景 A：运营人员从原料创建净料**
> 运营人员在原料清单看到"带皮五花肉"，点击"转净料"，在复合弹窗中依次创建"去皮五花肉片（出成率 85%）"、"去皮五花肉丝（出成率 82%）"，系统自动带出原料价格并计算净料单价，右侧实时展示该原料已创建的净料列表。

**场景 B：运营人员独立创建净料**
> 运营人员进入净料清单，点击"新增"，手动选择源原料，填写规格和出成率，系统自动计算净料单价。

**场景 C：运营人员查看原料衍生的净料**
> 运营人员查看原料详情，看到该原料已衍生出哪些净料规格，每个规格的成本和出成率。

## 三、核心痛点与价值主张

### 3.1 痛点清单

| 编号 | 痛点 | 当前现状 | 影响 |
|------|------|----------|------|
| P1 | 净料数据为 0 | 系统无净料档案 | 菜品 BOM 主料/辅料无法引用，排程切配工单无法生成 |
| P2 | 创建净料效率低 | 需手动填写单位、储存方式、分类等字段 | 大量重复操作，易出错 |
| P3 | 净料单价人工计算 | 无自动计算逻辑 | 运营人员需手动用计算器换算，易出错 |
| P4 | 原料与净料关系不可见 | 原料清单无"已衍生净料"提示 | 重复创建、遗漏规格 |

### 3.2 价值主张

| 痛点 | 解决方案 | 价值 |
|------|----------|------|
| P1 | 提供快捷入口批量创建 | 从原料一键转净料，省去手动选择源原料步骤 |
| P2 | 字段固定默认值（单位 500g、储存冷藏） | 减少 2 个字段的填写，表单更聚焦 |
| P3 | 出成率驱动自动计算净料单价 | 消除人工计算误差，成本数据即时准确 |
| P4 | 复合弹窗右侧展示已创建净料 | 实时可见，避免重复和遗漏 |

## 附录 A：数据完整定义表

### A.1 Schema 变更清单

#### `NetIngredient` 模型变更

```prisma
model NetIngredient {
  id                 Int             @id @default(autoincrement())
  code               String          @unique
  name               String
  sourceIngredientId Int             @map("source_ingredient_id")
  spec               String?
  yieldRate          Decimal         @map("yield_rate") @db.Decimal(5, 2)
  unitPrice          Decimal         @map("unit_price") @db.Decimal(10, 2)
  unit               String          @default("500g") @map("unit")
  l2Code             String          @map("l2_code")
  storage            String          @default("冷藏") @map("storage")  // ← 新增默认值
  createdAt          DateTime        @default(now()) @map("created_at")  // ← 新增
  updatedAt          DateTime        @updatedAt @map("updated_at")  // ← 新增
  deletedAt          DateTime?       @map("deleted_at")  // ← 新增（软删除）
  dishNetDetails     DishNetDetail[]
  sourceIngredient   Ingredient      @relation(fields: [sourceIngredientId], references: [id])
}
```

**变更说明**：
- `storage`：新增 `@default("冷藏")`，与 MVP1 固定值策略一致
- `createdAt`/`updatedAt`：补全时间戳，符合全局规范
- `deletedAt`：新增软删除，符合 AGENTS.md 数据库规范

#### `Ingredient` 模型（无变更，仅约束说明）

`latestRefPrice` 字段用于净料单价计算，**计算时假设其价格单位与净料单位（500g）一致**，不做额外单位换算（MVP1 约束）。

### A.2 API 变更清单

| 接口 | 方法 | 变更内容 |
|------|------|----------|
| `/api/net-ingredients` | GET | 新增 `?sourceIngredientId` 查询参数，用于复合弹窗右侧列表 |
| `/api/net-ingredients` | POST | ① 支持自动计算 `unitPrice`（前端不传时）② 保存 `createdAt` ③ `storage` 默认"冷藏" |
| `/api/net-ingredients/[id]` | PUT | 支持 `updatedAt` 自动更新 |
| `/api/net-ingredients/[id]` | DELETE | ① 检查 `dishNetDetails` 引用，若被引用则返回 400 + 错误信息 ② 改为软删除（`deletedAt = now()`） |
| `/api/ingredients/[id]` | GET | 新增 `includeNetIngredients` 参数，返回该原料关联的净料列表（用于原料详情页） |
| `/api/ingredients` | GET | 现有接口，无需变更（已有 `latestRefPrice` 字段） |

## 附录 B：核心设计机制

### B.1 净料单价自动计算机制

**触发时机**：表单中 `sourceIngredientId` 或 `yieldRate` 变化时自动计算。

**计算公式**：

```
unitPrice = sourceIngredient.latestRefPrice / (yieldRate / 100)
```

**示例**：

| 源原料 | 最新参考价 | 出成率 | 净料单价（自动计算） |
|--------|------------|--------|----------------------|
| 带皮五花肉 | ¥18.00/斤 | 85% | 18.00 / 0.85 = **¥21.18** |
| 带皮五花肉 | ¥18.00/斤 | 82% | 18.00 / 0.82 = **¥21.95** |

**边界处理**：

| 场景 | 处理方式 |
|------|----------|
| 源原料 `latestRefPrice` 为空 | `unitPrice` 不自动计算，留空等待用户手动输入 |
| `yieldRate` 为 0 | `unitPrice` 不自动计算，避免除零错误 |
| 用户已手动修改 `unitPrice` | 后续 `yieldRate` 变化时**不再自动覆盖**，但可提示用户"重新计算" |
| 源原料变更 | `unitPrice` 按新原料价格重新计算（若用户未手动覆盖） |

**前端实现**：用 `useEffect` 监听 `sourceIngredientId` 和 `yieldRate`，当两者都有值且 `unitPrice` 未被用户手动修改过时，自动计算并填充。

### B.2 复合弹窗交互规范

**布局**：`sm:max-w-[1200px]`，左右分栏（左 **40%** + 右 **60%**）。

**左侧表单（字段顺序）**：

| 字段 | 类型 | 必填 | 交互方式 | 只读条件 | 说明 |

- 已选择原料时：显示三个只读字段（一级分类、二级分类、来源原料），每行一个或并排展示
- 从原料转净料时：直接显示三个只读字段（自动带入，不可修改），不显示"选择"按钮
- 已选择原料后（非只读模式）：显示"更换原料"按钮，点击重新打开 IngredientSelectorDialog 可修改选择

1. 打开弹窗时，右侧列表加载 `GET /api/net-ingredients?sourceIngredientId=xxx`
2. 左侧保存成功后，右侧列表自动刷新
3. 右侧点击"编辑"，左侧表单切换为编辑态，右侧列表继续展示（编辑态不隐藏）
4. 保存成功后，编辑态重置为创建态（方便连续创建多个净料）
5. 右侧列表使用 `DataTable` 组件封装（与净料清单列表保持风格一致）

**弹窗交互规则**：
1. 打开弹窗时，右侧列表加载
2. 保存成功后，如果当前是创建态：重置表单为创建态（保留 sourceIngredientId 和 l2Code，方便连续创建），右侧列表刷新
3. 保存成功后，如果当前是编辑态：关闭弹窗，列表页刷新

### B.2.5 复合原料选择器（IngredientSelectorDialog）交互规范

**组件定位**：新增独立组件 `IngredientSelectorDialog`，用于在净料表单中一次性选择原料（连带一二级分类信息）。

**布局**：标准弹窗（`sm:max-w-[900px]` 或更大），左右分栏（左 **35%** + 右 **65%**）。

**左侧区域（分类选择）上下分区**：
- **上区：一级分类**（纵向平铺展示，非树结构）
  - 展示所有一级分类（`categories` 数组），每个分类为一个可点击的按钮/卡片
  - 默认**不选择任何一级分类**（全部未选中状态）
  - 点击某一级分类后，该分类高亮选中，下区展示对应的二级分类
  - 已选中的一级分类可以再次点击取消（回到默认状态，下区隐藏）
  - 样式：平铺按钮或卡片，选中态用 `bg-primary/10` + `border-primary`

- **下区：二级分类**（纵向平铺展示，非树结构）
  - **默认不展示**（未选择一级分类时，下区完全隐藏或显示空态提示"请选择一级分类"）
  - 选择一级分类后，下区展示该一级分类下的所有二级分类（`children` 数组）
  - 二级分类也是平铺按钮/卡片，点击后高亮选中
  - 已选中的二级分类可以再次点击取消
  - 选择二级分类后，**右侧表格自动过滤**显示该二级分类下的原料
  - 样式与上区一致，可用更紧凑的间距

**右侧区域（原料表格）**：
- 使用 `DataTable` 组件展示原料列表
- 表格字段：序号、编号、食材名称、商品名称、采购规格、采购单位、最新参照单价
- 默认展示**所有原料**（未选择分类时）或**当前二级分类下的原料**（已选择二级分类时）
- 表格行可点击，点击某行后：
  1. 该行高亮标记（如 `bg-primary/5`）
  2. 底部显示"已选择：XXX"确认文案
  3. 点击"确认选择"按钮关闭弹窗，回填表单
- 表格支持搜索（按名称/编号）

**选择后支持修改**：
- 表单中已选择原料后，显示"更换原料"按钮
- 点击重新打开 `IngredientSelectorDialog`
- 弹窗打开时，自动恢复到上次的选择状态（一级分类高亮、二级分类高亮、原料行高亮）
- 用户可以重新选择，点击确认后覆盖之前的选择

**从原料转净料时的特殊处理**：
- 不显示"选择来源原料"按钮，直接展示三个只读字段（一级分类、二级分类、来源原料）
- 不显示"更换原料"按钮（只读模式下不可修改）

**数据流**：
```
1. 用户点击"选择来源原料" → 打开 IngredientSelectorDialog
2. 用户点击一级分类 → 下区展示对应二级分类
3. 用户点击二级分类 → 右侧表格过滤显示该分类下的原料
4. 用户点击原料行 → 行高亮，底部显示已选原料
5. 用户点击"确认选择" → 关闭弹窗，回填 form.l1Code / form.l2Code / form.sourceIngredientId
6. 表单中显示已选原料信息，显示"更换原料"按钮
7. 用户点击"更换原料" → 重新打开弹窗，恢复上次选择状态
```

### B.3 删除校验机制

**硬删除改为软删除**：符合 AGENTS.md 规范。

**删除前引用检查**：

```
1. 查询 dishNetDetails.count({ where: { netIngId: id } })
2. 若 count > 0：返回 400，错误信息 "该净料已被 [N] 道菜品引用，无法删除"
3. 若 count = 0：执行 update({ deletedAt: new Date() })
```

## 附录 C：术语表

| 术语 | 英文 | 业务定义 |
|------|------|----------|
| 净料 | Net Ingredient | 原料经初加工后的规格化半成品，如"去皮五花肉片"，是菜品配方（BOM）和切配工单的基本单位 |
| 原料 | Raw Ingredient | 采购的基本食材，如"带皮五花肉"，是采购入库和库存管理的基本单位 |
| 出成率 | Yield Rate | 原料加工成净料后的重量保留比例，如 85% 表示 1 斤原料出 0.85 斤净料 |
| 净料单价 | Net Unit Price | 每 500g 净料的成本价格，计算公式：原料最新参考价 ÷ 出成率 |
| 源原料 | Source Ingredient | 净料加工前的原始原料，净料表通过 `sourceIngredientId` 关联 |
| 衍生净料 | Derived Net Ingredients | 由同一源原料加工出的多种净料规格，构成一对多关系 |
| 复合弹窗 | Composite Dialog | 左侧表单 + 右侧列表的复合布局弹窗，用于创建/编辑时参考已有数据 |
